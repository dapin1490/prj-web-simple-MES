## 최종 결정 (Decision)
<!-- 핵심만 빠르게 파악할 수 있도록 작성 -->

| 영역 | 최종 선택 | 버전/규칙 |
|---|---|---|
| Frontend | Vue.js, Pinia, Axios | Vue.js 3.x |
| Backend (애플리케이션) | Java, Spring Boot, Gradle | Java 17, Spring Boot 3.x |
| Backend (DTO/엔티티 매핑) | MapStruct(Mapper) | MapStruct (향후 `be-mes-project/build.gradle`에 추가 예정, 버전은 Spring Boot 3.x/Java 17과 호환 범위로 고정) |
| Database | MySQL | MySQL 8.x |

### 선택 이유

- Frontend
  - Vue.js 3.x는 빠른 개발과 유지보수가 용이하며, 현재 팀 구성원들이 공통적으로 사용 가능한 기술이기 때문에 선택되었음.
  - 단기간 내에 빠른 구현을 목표로 하기 때문에 신규 기술을 도입하기보다는 기존의 기술을 사용하기로 결정.
- Backend (애플리케이션)
  - 프론트엔드와 마찬가지로 단기간 구현을 목표로 팀원들이 공통적으로 사용 가능한 기술로써 채택됨.
  - 단 gradle의 경우 공통 기술은 아니지만, 증분 빌드를 사용하기 때문에 변동사항이 있을 때 빌드가 더 빠르고, 빌드 시간이 줄어들어 개발 생산성을 높일 수 있음. 또한 스크립트 관리가 용이하여 채택되었음.
- Backend (DTO/엔티티 매핑)
  - JDBC Repository 구현 방식과 달리 mapper를 사용하여 코드 가독성과 유지보수성을 높이기 위해 채택됨.
  - 인터페이스 정의만으로 구현 코드를 자동으로 생성할 수 있어 개발 생산성을 높일 수 있음.
- Database
  - 프론트엔드와 마찬가지로 단기간 구현을 목표로 팀원이 공통으로 사용 가능한 기술로 채택됨.

## 대안과 트레이드오프
<!-- 최종 결정된 사항을 포함해 최소 2개 방안을 비교 -->

| 옵션 | 장점 | 단점 | 선택 여부 |
|---|---|---|---|
| Gradle 빌드 | 코드 업데이트에 대한 빠른 빌드 | 비숙련 기술 | O |
| mapper 방식 | 인터페이스 정의만으로 빠른 구현 생성됨 | 외부 라이브러리 의존성 존재 | O |
| JDBC Repository 방식 | 외부 라이브러리 의존성 낮음 | SQL 쿼리와 자바 구현 코드가 분리되지 않고 하나의 클래스에 작성되어 가독성이 떨어지고, 직접 모든 사항을 구현해야 하므로 개발 생산성 저하될 수 있음 | X |
| Maven 빌드 | 팀원 공통 기술 | gradle에 비해 코드 변경 후 빌드 시간이 오래 걸림 | X |
| SQLite | DB 용량이 작아 저장공간을 절약할 수 있고, 별도의 DBMS를 필요로 하지 않기 때문에 메모리를 절약할 수 있음 | Gradle과 마찬가지로 비숙련 기술이기 때문에 신규 도입 리스크 존재함 | X |
| 도커 | 일반적인 배포 방식 중 하나로 많이 사용되고 있기 때문에 현행 최신 기술로써 고려됨, 본 프로젝트 이후 예정된 강의 존재함 | 비숙련 기술로써 아직 모르는 상태로 도커 배포를 미리 고려한 코드를 작성하는 것은 리스크가 크다고 판단 | X |

## 기대 효과와 리스크 (Consequences)
### 기대 효과
- 팀원들이 일관된 방식으로 개발할 수 있음

### 리스크 및 대응
- 리스크
  - 팀원들이 비숙련 기술을 사용하게 될 경우 개발 생산성이 저하될 수 있음.
  - mapper 방식의 경우 외부 라이브러리 의존성 존재로 인해 라이브러리의 업데이트나 deprecated 등의 문제에 영향을 받을 수 있음.
- 대응
  - 생성형 AI와 같은 도구를 활용해 빠른 학습과 적응을 도모하고, 설계 문서를 최대한 꼼꼼하게 작성하여 설계와 구현의 일관성과 정합성을 유지함.
  - 지속적인 유지보수로 의존성 관리와 업데이트 수행.
