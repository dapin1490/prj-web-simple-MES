## 심플 MES 통합 기능 요구 명세서 (SRS v2.2)

### 1. 프로젝트 개요 (Overview)

- 목적: 제조 현장의 수주 관리, 생산 계획, 실시간 공정 감시, 품질 분석을 아우르는 데이터 중심의 MES를 구축함.
- 설계 철학: ISA-95 표준의 Level 3 기능을 지향하며, 실제 산업 현장의 데이터(KAMP 데이터셋)를 기반으로 동작하는 실무형 시스템을 구현함.
- 핵심 지표: 생산 진척도(Progress), 설비 가동 효율(OEE), 품질 합격률(Pass Rate).

### 2. 기술 스택 (Technology Stack)

- Backend: Java Spring Boot 3.x.
- Frontend: Vue.js 3.x.
- Database: MySQL 8.x.
- Communication: REST API, STOMP 기반 WebSocket (실시간 데이터 전송).

### 3. 데이터 통합 및 정합성 전략 (Data Integration Strategy)

본 시스템은 런타임 시의 별도 데이터 가공 로직을 포함하지 않으며, 사전 전처리된 통합 데이터셋을 기반으로 동작합니다.

### 3.1. 데이터 소스 구성

- 수주 원천: 에스윈텍 고객사 주문량 데이터셋 (약 93,509건 기록).
- 생산 실행: 의류 공정최적화 데이터셋 (로트별 2,127건 및 트렌드 211만 건).
- 품질 검증: 의류 공정 CCM 측정값 (434개 로트에 대한 정밀 측정치).

### 3.2. 정합성 처리 지침 (Data Consistency Rules)

- 수량 동기화 (Pre-overwritten): 제공되는 데이터셋은 에스윈텍 주문량(`day_order`)이 의류 공정의 일일 생산 합계치($\sum$ `염색 가동 길이`)와 이미 일치하도록 전처리가 완료된 상태입니다.
- 날짜 동기화 (Pre-aligned): 모든 주문 및 생산 레코드는 의류 공정 실측 기간인 2022년 1월~10월 범위 내로 이미 시점이 조정되어 있습니다.
- 제품 정보 (Pre-mapped): 비공개된 제품 코드는 의류 공정 코드(`JOB_CD`) 기반의 가상 제품명으로 이미 치환되어 제공됩니다.

### 4. 데이터베이스 물리 스택 설계 (Physical DB Schema)

| 테이블명 | 핵심 컬럼 | 설명 |
| --- | --- | --- |
| Products | `product_id`(PK), `name`, `safety_stock` | 제품 마스터 정보. |
| SalesOrders | `order_id`(PK), `product_id`(FK), `order_qty` | 동기화된 일일 주문 정보. |
| WorkOrders | `wo_id`(PK), `order_id`(FK), `planned_qty` | 생산 로트(LOT) 단위 작업 지시. |
| ProductionLogs | `log_id`(PK), `wo_id`(FK), `temp_pv`, `timestamp` | 1분 주기 설비 시계열 로그. |
| Inspections | `insp_id`(PK), `wo_id`(FK), `color_de` | 품질 검사 및 색차 측정 결과. |

### 5. 모듈별 세부 기능 요구사항 (Detailed Requirements)

### 5.1. 수주 및 생산 계획 모듈 (Order & Planning)

- 데이터 로드: 백엔드는 전처리된 `SalesOrders` 데이터를 DB에 로드하여 수주 현황을 관리합니다.
- 작업 지시 발행: 이미 수치가 맞춰진 주문 데이터를 기반으로 현장의 `WorkOrders`(로트 단위)를 생성하고, 데이터셋에 정의된 투입 사양을 그대로 할당합니다.
- 수주 잔량 관리: 에스윈텍 기반의 '당월총주문량'과 '총주문잔량' 데이터를 비교하여 미납 주문의 우선순위를 시각화함.
- 생산 트리거: 시스템 내 '현재고량'이 '안전재고' 수치 이하로 하락할 경우 생산 계획 대기 리스트에 해당 품목을 자동 등재함.
- 작업 지시(Work Order) 발행: 확정된 생산 계획에 따라 의류 공정의 'LOT번호'를 생성하고, '투입중량(kg)'과 '염색길이(m)' 사양을 할당함.

### 5.2. 실시간 공정 감시 모듈 (Execution & Monitoring)

- 설비 상태 매핑: 각 '설비번호'별로 현재 가동 중인 'LOT번호'를 실시간 매칭하여 현황판에 표시함.
- 진척률 산출 로직: 공정 진행 시간 데이터를 활용하여 다음의 수식에 따라 진행률($P$)을 실시간 계산함.
    
    $$P = \frac{t_{current}}{t_{total}} \times 100$$
    
- 환경 변수 시각화: '진행온도' 및 '포속 1~4'의 변화량을 실시간 꺾은선 그래프(Line Chart)로 제공함.

### 5.3. 생산 실적 및 품질 리포트 모듈 (Reporting & Quality)

- 품질 자동 판정: CCM 측정 데이터의 '염색색차 DE' 값을 기준치(예: 1.0)와 비교하여 합격(Pass) 또는 불합격(Fail)을 자동 판정함.
- 최종 실적 집계: 생산 완료 시 실제 '투입중량/길이' 및 '공정진행시간(%)'을 요약하여 기록함.
- 보고서 자동 생성: 수집된 공정 로그와 품질 데이터를 조인하여 표준화된 양식의 생산 검사 성적서를 생성함.

### 6. 사용자 플로우 (User Flow)

1. 로그인 및 권한 확인: 사용자는 시스템 접속 시 관리자(Admin) 또는 작업자(Operator) 권한을 부여받음.
2. 수주 확인 및 지시: 관리자가 수주 현황 및 재고 부족 품목을 확인하고 작업 지시(Work Order)를 발행함.
3. 생산 시작 및 감시: 작업자가 생산 시작 버튼을 누르면 시뮬레이터가 작동하며, 현황판에 실시간 진행 온도와 포속 데이터가 출력됨.
4. 품질 검사: 공정 종료 후 시스템은 CCM 측정값을 로트별로 자동 결합하여 판정 결과를 노출함.
5. 보고서 출력: 최종 결과 확인 후 표준 양식의 리포트를 생성하여 파일(PDF/HTML)로 저장함.

### 7. 핵심 기술 로직: 시뮬레이터 (Simulator)

백엔드 시뮬레이터는 복잡한 데이터 변환 로직 없이, 이미 정제된 로그를 순차적으로 적재하는 역할에 집중합니다.

- 동작 원리: Spring Boot의 `@Scheduled` 어노테이션을 사용하여 전처리 완료된 통합 로그 파일의 로우(Row) 데이터를 MySQL의 `ProductionLogs` 테이블로 순차적 삽입(Insert)합니다.
- 무결성 보장: 시뮬레이터가 읽는 모든 로그는 이미 `WorkOrders` 테이블의 고유 번호(`wo_id`)와 정합성이 확보된 상태이므로, 별도의 유효성 검사 로직을 간소화합니다.
- 프론트엔드 연동: 데이터 적재와 동시에 WebSocket(STOMP)을 통해 Vue.js 클라이언트로 변경 사항을 Push하여 실시간 차트를 갱신합니다.

### 8. 명세와 구현 시 참고 (스키마·API 우선)

- **물리 스키마의 정본**: 테이블·컬럼명·타입·관계는 `docs/data-schema-definition.md` **§2**를 따른다. 본 문서 §4의 표는 요약이며, 구현·마이그레이션·REST 필드는 §2와 일치시킨다.
- **식별자 정합**: 전처리 데이터에서 `WorkOrders.wo_id`(의류 `PRODT_ORDER_NO`)와 `ProductionLogs.wo_id`(의류 `LOT_NO`)는 동일 로트를 가리키도록 이미 매핑되어 있다는 전제를 둔다. 상세는 `docs/data-generation-report.md` 및 정합성 지침을 참고한다.
- **범위가 넓은 요구**: OEE, 로그인·권한, PDF/HTML 내보내기, 투입중량·염색길이 등은 목표 기능을 기술한 것이다. 현재 `docs/data-schema-definition.md`·`docs/api-details.md`에 없는 항목은 **추가 명세를 문서에 반영한 뒤** 구현한다.
- **포속(속도) 필드**: §5.2의 '포속 1~4'는 현장 데이터 표현을 넓게 가리킨다. 스키마·API·WebSocket 샘플에 명시된 공정 속도는 `ProductionLogs.speed`(원천 `TRD_SPEED1`) **단일 필드**로 시작하고, 추가 축이 필요하면 스키마와 `api-details.md`를 함께 갱신한다.
