# 협업 컨벤션

## 1. 작업 흐름

<!--
이 항목에서 정하는 것:
- 이슈를 언제·어떤 단위로 만드는지(작업 크기, 라벨 사용 여부)
- 브랜치 이름 규칙과 기본 전략(예: main에서 분기, 보호 브랜치 여부)
- PR을 열기 전에 갖출 것(최소 리뷰, CI, 연결 이슈 번호)
- 머지 방식(스쿼시·머지 커밋·리베이스 중 무엇을 쓸지)
- 설계·스키마·API 문서와 구현이 어긋날 때의 순서(문서 먼저 수정 후 코드 등)
-->

> 본 절의 내용은 [COLLABORATION.md](../COLLABORATION.md) 문서의 1, 7, 8절과 [.github/WORKFLOW.md](../.github/WORKFLOW.md)를 일부 참조한다.
> 단, 협업 규칙의 단일 기준은 이 문서이며, 위 문서들은 참고용으로만 활용한다. 커밋/브랜치/PR 규칙 등에서 표현 예시나 세부 규칙이 상충하는 경우에는 이 문서의 기준을 우선 적용한다. 필요 시 참조 문서도 이 문서의 기준에 맞추어 함께 정비한다.

- 이슈 생성 시점: 대략적인 기능 단위 개발 전, 무슨 기능을 개발할 것인지 선언하는 형식으로 이슈 생성
  - 라벨은 크게 프론트엔드, 백엔드, 데이터(백엔드 이외), 문서화로 구분되며, 필요한 경우 다른 라벨(깃허브 기본 제공)을 사용할 수 있다.
  - 라벨의 수는 제한하지 않는다.
- 브랜치 이름 규칙: `feature/<이슈번호>-<짧은-기능명>`
  - main은 보호된 브랜치로, 직접 커밋할 수 없고 PR을 통해서만 merge할 수 있다.
  - 새 브랜치를 분기할 때에는 특별한 이유가 없는 한 최신의 main으로부터 분기한다.
  - 브랜치 이름은 소문자로 작성하며, 공백은 -로 대체한다.
  - 브랜치 이름은 구현하고자 하는 기능 단위를 구분할 수 있도록 작명한다.
- PR 규칙
  - 기본적으로는 이슈에서 선언한 TODO가 모두 완료되었고, 정합성이 확인되었으며, 오류 없이 동작함이 확인된 후에 PR을 생성한다.
  - PR 내용은 사전에 정의된 템플릿([.github/pull_request_template.md](../.github/pull_request_template.md))에 맞추어 작성한다.
  - PR 템플릿에 제시된 항목은 가능한 한 모두 작성하고, 채울 수 없는 내용은 채우지 않은 사유를 대신 작성한다.
  - PR은 merge만 사용한다. 특별한 사유가 없는 한 rebase나 squash는 사용하지 않는다.
- 설계 변경 시 docs 내의 명세 문서를 먼저 수정하여 설계의 일관성을 유지하고, 이후 문서에 따라 개발한다.
  - 새로 변경된 설계가 있을 경우 그와 관련된 문서를 전수 검토하여 수정이 필요한 문서를 모두 업데이트한다.
- 개발 워크플로우 ([.github/WORKFLOW.md](../.github/WORKFLOW.md) 참조)
  1. 이슈 생성: 구현하고자 하는 기능에 대한 TODO를 선언하여 이슈를 생성한다. 이에 지정된 [이슈 템플릿](../.github/ISSUE_TEMPLATE/todo.md)을 사용하며, 제시된 항목을 모두 채우도록 한다. 선택 항목은 제외 가능하다.
  2. 브랜치 생성: 1에서 생성한 이슈로부터 브랜치를 만든다.
  3. 커밋: 브랜치에서 커밋을 생성한다.
  4. PR 생성: 3에서 생성한 커밋을 기반으로 PR을 생성한다.
  5. PR 리뷰: PR을 리뷰하고, 문제가 있는 경우 수정을 요청한다. 브랜치 룰에 의해 copilot에게 자동으로 리뷰가 요청되며, 이를 참고하여 필요한 경우 수정한다. 문제가 없다면 approve한다.
  6. merge: approve가 완료된 PR을 main으로 merge한다.

## 2. 커밋 메시지

<!--
이 항목에서 정하는 것:
- Conventional Commits 사용 여부와 허용 타입 목록(feat, fix, docs 등)
- 제목·본문·이슈 참조 규칙(한글·영문, 길이 가이드)
- 스쿼시 머지 시 최종 커밋 한 줄을 어떻게 맞출지(템플릿이 있으면 참조)
-->

> 본 절의 내용은 [COLLABORATION.md](../COLLABORATION.md) 문서의 7절을 일부 참조한다.

- 커밋 메시지는 Conventional Commits 형식을 사용한다.
- 커밋 메시지는 다음과 같은 형식으로 작성된다: `태그(영역): 변동사항 설명 #이슈번호`
- 태그는 `feat`, `fix`, `docs`, `chore`, `refactor`로 구분되며, 공통이면 우선하는 것 하나만 작성한다.
- 영역은 프론트엔드(`fe`), 백엔드(`be`), 데이터(`data`, 백엔드 이외)로 구분되며, 공통이면 우선하는 것 하나만 작성한다.
- 태그와 영역은 소문자로 작성한다.
- PR 시 최종 커밋은 깃허브 기본값으로 사용하며, 직접 작성할 경우 최소한 PR 번호와 merge된 브랜치를 명시한다. 예: `Merge pull request #PR번호 from 작성자/브랜치명`

## 3. 코드 품질

<!--
이 항목에서 정하는 것:
- 프론트엔드·백엔드 각각 사용하는 린터·포맷터와 설정 파일 위치
- PR 전에 로컬에서 돌릴 최소 검사(명령어 또는 npm 스크립트 이름)
- 폴더·파일·컴포넌트 네이밍은 예시 몇 가지로만 적어도 된다(전체 스타일 가이드는 선택)
-->

> 본 절의 내용은 [COLLABORATION.md](../COLLABORATION.md) 문서의 6절을 일부 참조한다.

- 린터
  - 프론트엔드는 ESLint를 사용한다.
  - 백엔드는 현재 Checkstyle 등의 린터를 도입하지 않았으며, 추후 도입 시 설정 파일 위치와 실행 방법(Gradle task/CI)을 별도로 문서화한다.
  - 데이터는 데이터 스키마 정의 문서([docs/data-schema-definition.md](../docs/data-schema-definition.md))를 참고하여 작성한다.
- PR 전 로컬에서 정합성을 검토하고 오류 없이 실행되는 것을 확인한다.
- 폴더명은 소문자로 작성하며, 공백은 `_`로 대체한다. `.github/ISSUE_TEMPLATE`은 이 규칙에 대해 예외로 간주한다.
- 파일 및 컴포넌트의 이름은 각 언어 및 프레임워크의 컨벤션을 따른다. 단, docs의 문서는 케밥 케이스로 작성한다.
  - 예:
    - 프론트엔드: `src/components/Button.vue`
    - 백엔드: `src/main/java/com/example/demo/controller/UserController.java`
    - 데이터: `scripts/data_preprocess.py`
    - docs: `docs/data-schema-definition.md`

## 5. 역할과 소통

<!--
이 항목에서 정하는 것:
- 코드 리뷰는 누가·몇 명 승인인지(2인 팀이면 상호 리뷰 등)
- 질문·결정·막힌 일을 어디에 남길지(이슈 코멘트, 디스커션, 채널)
- 회의나 스탠드업이 있다면 주기, 없다면 비동기만으로 어떻게 맞출지
-->

> 본 절의 내용은 [COLLABORATION.md](../COLLABORATION.md) 문서를 일부 참조한다.

- 코드 리뷰: 두 명의 개발자가 모두 업무 가능한 상태인 경우 상호 리뷰를 반드시 하며, 한 명 이상이 업무 불가능한 경우 copilot 리뷰로 대체한다.
- 진행 중 문의 사항이 있을 경우, 진행 중이던 브랜치를 생성한 이슈에서 서브 이슈를 생성하여 문의를 남긴다. 이슈에 대한 문의는 이슈 내의 댓글로 남긴다. 디스커션은 사용하지 않는다. (소규모 팀이기 때문에 소통 창구를 너무 다양하게 갖는 것은 비효율적이라 판단)
- 평일 오전 9시부터 오후 6시(공식 수업 인정 시간 한정) 사이에는 별다른 일정 상의 사유가 없는 한 대면 작업으로 진행하며, 비대면이 필요한 경우 적절한 수단(카톡, 온라인 회의 등)을 사용한다.

## 6. 아키텍처 결정 기록 (ADR)

<!--
이 항목에서 정하는 것:
- ADR을 두는 디렉터리 경로(예: docs/adr)
- 파일 이름 규칙(예: 0001-짧은-제목.md)
- 각 ADR에 넣을 최소 뼈대(맥락, 결정, 결과, 대안 요약 등)
- “언제 ADR을 새로 쓰는지” 기준(스택 변경, 경계 변경 등)
-->

> 본 절의 내용은 [COLLABORATION.md](../COLLABORATION.md) 문서를 일부 참조한다.

- ADR 위치: [`docs/adr`](../docs/adr)
  - 파일 이름 규칙: `날짜-시간-짧은-제목.md` (예: `260414-1633-setup-convention.md`)
  - 각 ADR에 넣을 최소 뼈대: 맥락, 결정, 결과, 대안 요약(왜 선택하지 않았는지 포함) 등
  - 언제 ADR을 새로 쓰는지: 스택 변경, 경계 변경, 규칙 변경 등
- ADR은 프로젝트의 중요한 결정을 기록하고, 이후 변경 시 참고하여 일관성을 유지하는 데 사용한다. 특히 규칙 변경 시 참고하여 일관성을 유지하는 데 사용한다.

## 7. Pull Request 템플릿 및 작성 규칙

<!--
이 항목에서 정하는 것:
- PR 본문에 반드시 포함할 항목(변경 요약, 관련 이슈, 테스트 방법, UI 변경 시 스크린샷 등)
- 실제 GitHub PR 템플릿 파일 위치(예: .github/pull_request_template.md)와 이 문서와의 관계
- 초안(Draft) 사용 여부, 리뷰 요청 타이밍
-->

> 본 절의 내용은 [COLLABORATION.md](../COLLABORATION.md) 문서와 [.github/pull_request_template.md](../.github/pull_request_template.md)를 일부 참조한다.

- PR 본문에 반드시 포함할 항목:
  - 변경 요약
  - 관련 이슈
  - 테스트 방법
  - UI 변경 시 스크린샷 등
- 개발 과정에서 merge 이전에 리뷰가 필요한 작업이 생길 경우 Draft PR을 생성하여 리뷰를 요청한다.
- 리뷰는 PR 생성 후 작성자가 copilot의 리뷰 검토를 완료한 후 리뷰어에게 요청한다. 리뷰어는 리뷰를 완료한 후 approve한다.
